## 1. テニスのセオリー調査 — 戦略と深みの源泉

実テニスで「強さ」「面白さ」を生んでいる要素を、ゲームの戦略軸に落とせる形で整理する。

### 1.1 配球(コート上の幾何)— 「相手を動かして空きを作る」

- **オープンコート理論**: 相手を一方に追い出すと逆サイドが空く。1球で決めず、
  「振って → 空けて → 逆を突く」の2〜3球の組み立てで点を取る。
- **サーブ+1(Serve plus one)**: 良いサーブで甘い返球を誘い、次の1球(フォア)で
  攻め切る。プロのポイントの大半はこの3球目までで決まる。
- **ワン・ツーパンチ**: 1球目で相手を動かし、2球目で逆 or 同サイドに「決め」を入れる。
- **ワイド→ボディ→T(サーブのコース)**: サーブの的を散らして読まれないようにする。
- **逆を突く / ワンモアステップ**: 相手が動き出した方向と逆へ打つ(wrong-footing)。
- **アングル(短く鋭い角度)**: 相手をコート外へ追い出し、次球のオープンを最大化。
- **深さ(深い球)**: 深い球は相手を押し下げ、攻撃の起点を奪う。浅い球は叩かれる。

→ **既存ゲームへの示唆**: 左右(A/D)・前後(W/S)・5ショット種で既に表現されている。
  足りないのは「組み立てが報われる仕組み」(=相手を動かした分だけ次が有利になる連鎖)。
  §4 の「ポジショニング/オープンコート可視化」「連続で振った時のスタミナ削り」で補強。

### 1.2 緩急・高低・左右(3次元の揺さぶり)

- **緩急(ペース変化)**: 速い球の中に遅い球(スライス/ドロップ)を混ぜるとタイミングを崩せる。
- **高低(バウンドの高さ)**: 高く跳ねる重いトップスピン(順回転)は打点を上げさせ、
  低く滑るスライスは沈ませる。打点を狂わせるのが目的。
- **左右(コース)**: §1.1 の配球。

→ 既存の §4.5(打点の高さ × 文脈)、§4.6(差し込まれ)、サーブ種別で**ほぼ実装済み**。
  ペルソナはこの「どの揺さぶりが得意か」を個性化する装置になる(§3)。

### 1.3 コートポジションとプレイの型

- **ベースライナー**: 後方で打ち合う。安定とパワーの勝負。
- **ライジング/アーリーボール**: バウンド直後を叩いて相手の時間を奪う(攻撃的)。
- **サーブ&ボレー / ネットプレー**: 前に詰めて角度で決める。ボレーとパッシングの読み合い。
- **カウンターパンチャー**: 守ってしのぎ、相手のミスと無理を誘う。脚力と粘りが武器。

→ これが **プレイスタイルの類型 = ペルソナの背骨**(§3 で6タイプ化)。

### 1.4 プレッシャーと勢い(メンタル)

- **ブレークポイント / ゲームポイント**の重み。大事な場面で崩れる/勝負強い差。
- **モメンタム(勢い)**: 連続得点で波に乗る、流れが変わる。

→ **新規追加候補**(§4)。AI とプレイヤー双方に「プレッシャー時の品質変動」を入れると
  ドラマ性が増す。ペルソナの**メンタル/勝負強さ**を隠しパラメータにできる。

### 1.5 体力配分

- 長いラリー・長いマッチでスタミナが効く。グラインダーは相手を消耗させて終盤に勝つ。

→ 既存のスタミナ系で実装済み。ペルソナでスタミナの最大値・回復・消耗を差別化する。

## 2. 既存システムとのマッピング(何が既にあり、ペルソナで何を動かすか)

ペルソナの能力は**新システムを足さず、既存の調整パラメータへ倍率を掛ける**だけで表現できる。
これは「`constants.ts` に集約、マジックナンバーを散らさない」規約とも合致する。

| セオリー要素 | 既存パラメータ(constants.ts / types.ts) | ペルソナで動かす軸(§3.1) |
|---|---|---|
| 移動・カバー範囲 | `WALK_SPEED` `SPRINT_SPEED` `REACH` `MOVE_ACCEL` | スピード |
| 体力配分 | `STAMINA_MAX` `*_DRAIN` `*_REGEN` `STAMINA_LOW_THRESHOLD` | スタミナ |
| サーブの威力/安全 | `SERVE_SPEED_*` `SERVE_TYPE_PARAMS` `faultNoiseMul` | サーブ |
| 強打・フラット | `SHOT_PARAMS.speed` `CHARGE_POWER_GAIN` `HIGH_CONTACT_SPEED_GAIN` | パワー |
| 安定性・ミス耐性 | `AIM_NOISE_R` `netMargin` `RETURN_WEAKNESS_*`(差し込まれ) | スピン/安定 |
| タッチ・ネット・角度 | `drop`/`lob`/`slice` 誤差、`HIGH_TOPSPIN_ANGLE_BONUS`、`PACE_TOUCH_PENALTY` | 技巧 |
| 返球(リターン) | `RETURN_WEAKNESS_*` `RETURN_CHARGE_MITIGATION` | (技巧/安定に内包) |
| AI の挙動 | `AIProfile`(reactionDelay/speedScale/aggressiveness/blunderRate) | 難易度 × ペルソナ |

**重要な設計分離**: 「難易度(easy/normal/hard)」と「ペルソナ」は直交させる。
- **難易度** = 相手 AI の総合的な"上手さ"(反応速度・読み・凡ミス率)。
- **ペルソナ** = プレイスタイルと能力の"配分"(誰がやっても穴と武器がある)。

これにより「Hard のサンブラント(サーブ最強)」「Easy のナダウ(粘るが遅い)」のような
組み合わせが生まれ、相性ゲームになる。

## 3. プレイヤーペルソナ(本命提案)

ペルソナは単なる**能力パラメータの束**ではなく、**プロファイル**として3層で定義する:

1. **能力(§3.1〜3.2)** — 6軸レーダーで表すプレイ性能。
2. **身体(§3.6)** — 身長・体格・利き手などの身体的特徴。識別の核であり、
   一部は軽いゲーム挙動にも繋がる(高身長=サーブ打点が高い、左利き=フォア/バック反転 など)。
3. **外見(§3.7)** — 3Dモデルの見た目(体格シルエット+識別フィーチャー)。
   自/敵は **1Pカラー(青系)/2Pカラー(赤系)** で塗り分け、シルエットと
   アクセントでペルソナを判別できるようにする。

### 3.1 能力パラメータ — 6軸(レーダーチャート)

ひと目で個性が分かるよう **6軸 ・ 各 1〜5 段階**(★の数)で表現する。
6軸は正六角形レーダーで描くと視認性が高い。各ペルソナは合計値をおおむね揃え、
**全員に明確な武器と弱点がある**ようにバランスする(誰も上位互換にしない)。

| 軸 | 意味 | 強いと何が起きるか(ゲーム内) |
|---|---|---|
| **サーブ** | サーブの速度と精度 | サーブ初速↑・フォルト率↓・1st の威力↑ |
| **パワー** | ストロークの球威 | フラット/トップスピン初速↑・チャージ威力↑・高打点で更に伸びる |
| **スピン/安定** | 回転量とミスのしにくさ | 狙い誤差↓・ネット越えマージン↑・速球に差し込まれにくい(粘り) |
| **スピード** | 機動力とリーチ | 移動速度↑・スプリント↑・`REACH`(届く範囲)↑=拾える |
| **スタミナ** | 体力の総量と持続 | `STAMINA_MAX`↑・消耗↓・回復↑=終盤も精度を保つ |
| **技巧** | タッチ・ネット・角度 | ドロップ/ロブ/スライス/アングルの誤差↓・速球リターンが上手い |

> レーダーUIの実装は §3.4。チャート上では時計回りに
> サーブ → パワー → スピン/安定 → スタミナ → 技巧 → スピード の順に配置すると、
> 右側=攻撃(サーブ・パワー)、左側=守備・持久(スタミナ・技巧・スピード)で
> プロファイル形状の意味が読み取りやすい。

### 3.3 相性(バランスの意図)

おおまかな三すくみで、選択に意味を持たせる:

- **攻撃型(サンブラント/フェデルン/アガチ)** は短期決戦で**グラインダー(ナダウ/ジョコヴィン)**
  を押し切れるが、安定して凌がれると消耗して失速。
- **グラインダー/カウンター(ナダウ/ジョコヴィン)** は長いラリーで攻撃型を削るが、
  自分から決められず**スピード/技巧型(ニシゴオリ)**の緩急・前後に翻弄される。
- **スピード/技巧型(ニシゴオリ)** は機動力で誰からも拾うが、**スタミナが弱点**で
  長期戦・高い打点の強要に弱い。

→ プレイヤーは「自分の型 × 相手の型」を読んで戦術を変える(攻め急ぐ/粘る/走らせる)。

### 3.4 ゲーム内パラメータへの変換(実装方針)

能力値(1〜5)から**倍率を導出**して既存定数に掛ける。新システムは増やさない。
変換式は `constants.ts` に集約する(マジックナンバーを散らさない規約)。例(初期案・要チューニング):

```ts
// 能力値 r ∈ {1..5} → 倍率。r=3 を基準(=1.0付近)にする。
serve   : serveSpeedMul = 0.92 + 0.04·r          // r5→1.12  r1→0.96
          faultNoiseMul = 1.30 − 0.12·r          // r5→0.70(正確)  r1→1.18
power   : shotSpeedMul  = 0.90 + 0.045·r         // r5→1.125 r1→0.945
          chargeGainMul = 0.80 + 0.10·r
spin    : aimNoiseMul   = 1.30 − 0.12·r          // r5→0.70(ミス少)
          netMarginMul  = 0.80 + 0.10·r          // r5→安定
          returnSolidMul= 1.30 − 0.12·r          // 差し込まれ(mishit)を軽減
speed   : moveSpeedMul  = 0.88 + 0.06·r          // r5→1.18  r1→0.94
          reachMul      = 0.92 + 0.035·r         // 拾える範囲
stamina : staminaMaxMul = 0.70 + 0.12·r          // r5→1.30(=130)
          drainMul      = 1.20 − 0.10·r          // r5→消耗0.70
          regenMul      = 0.80 + 0.10·r
finesse : touchNoiseMul = 1.30 − 0.12·r          // drop/lob/slice/angle の誤差
          returnTouchMul= 1.20 − 0.10·r          // 速球リターンの上手さ
```

> 値は「r=3 で現状とほぼ同じ」「最強でも 1.1〜1.3 倍程度」に抑え、
> 難易度(AIProfile)との二重スケールで過剰にならないようにする。要 §4 の検証スクリプトで調整。

**型の追加(`types.ts`)** — 依存ルールに従い契約だけ定義:
```ts
export type PersonaId =
  | 'sambrant' | 'agachi' | 'jokovin' | 'nishigoori' | 'nadau' | 'federun'

export interface PersonaRatings {  // 各 1..5(レーダー描画にそのまま使う)
  serve: number; power: number; spin: number
  speed: number; stamina: number; finesse: number
}

// 身体的特徴(§3.6)。識別の核であり、一部は軽いゲーム挙動にも繋がる。
export interface PersonaPhysique {
  heightM: number                    // 身長(m)。モデルの縦スケールと打点高に影響
  build: 'slim' | 'athletic' | 'stocky'  // 体格(手足・胴の太さスケール)
  handedness: 'right' | 'left'       // 利き手。left はフォア/バックの判定を左右反転
}

// 外見(§3.7)。チームカラー(1P/2P)とは独立した「ペルソナの識別子」。
export interface PersonaAppearance {
  hair: 'short' | 'bald' | 'headband' | 'long' | 'cap'  // シルエット識別フィーチャー
  sleeves: 'sleeved' | 'sleeveless'  // 袖の有無(シルエット)
  accent: number                     // 小物(ヘッドバンド/リスト/シューズ/ガット)のhex。
                                     // 青/赤に紛れない色(白・黄・緑・橙・黒等)を選ぶ
}

export interface Persona {
  id: PersonaId
  name: string          // 表示名(もじり)
  archetype: string     // 一言キャッチ(例「ビッグサーバー」)
  blurb: string         // 武器と弱点の短い説明
  ratings: PersonaRatings
  physique: PersonaPhysique
  appearance: PersonaAppearance
}
```
- `PLAYER_PERSONAS: Record<PersonaId, Persona>` を `constants.ts` に定義(§3.2 の値)。
- 倍率は `personaModifiers(ratings)` で都度導出 or 事前計算。`player.ts` / `ai.ts` /
  `shot.ts` は**注入された倍率を受け取るだけ**にして、依存ルール(他モジュール非依存)を保つ。
  → `main.ts` が選択結果を読み、各コントローラ・ソルバへ倍率を注入する結線役を担う。
- AI がペルソナを持つ場合、`AIProfile` に対し: `aggressiveness += f(serve,power)`、
  `speedScale ×= moveSpeedMul`、`blunderRate ×= f(spin,stamina)` のように**難易度の上に重ねる**。

### 3.5 プレイヤータイプ選択UI(タイトル画面)

**要件**(ユーザー要望):
- プレイ開始時に **自分のタイプ**と**相手のタイプ**をそれぞれ選択できる。
- それぞれ **ランダム選択**も可能(「おまかせ」)。
- レーダーチャートで能力をひと目で比較できる。

**UI 案**(DOM オーバーレイ、`ui/` に追加。外部アセットなしの規約に従い CSS+canvas/SVG で描画):

- 既存 `MatchConfig`(difficulty, gamesToWin)に `playerPersona` `opponentPersona` を追加し、
  `UIHandlers.onStart(config)` 経由で `main.ts` に渡す。

**実装タッチポイントまとめ**:
1. `types.ts`: `PersonaId` `Persona` `PersonaRatings` `PersonaPhysique`
   `PersonaAppearance` 追加、`MatchConfig` に 2 フィールド追加。
2. `constants.ts`: `PLAYER_PERSONAS`(能力+身体+外見)と倍率導出式、チームパレット定義。
3. `ui/menu`: 2つのピッカー + レーダー描画 + ランダムボタン。
4. `ui/` レーダー描画コンポーネント(canvas/SVG、外部ライブラリなし)。
5. `main.ts`: 選択結果を読み、player/ai コントローラとショットソルバへ倍率注入。
   `CharacterEntity` 生成時に身体・外見・チーム(1P/2P)を渡す。
6. `player.ts` / `ai.ts` / `shot.ts`: 注入された倍率を既存計算に乗算(契約は注入のみ)。
   左利きはフォア/バック判定の左右反転を `player.ts` のスイング側決定に適用。
7. `render/entities.ts`: `CharacterEntity` を**体格パラメータと配色を受け取る形に
   パラメータ化**(§3.7)。現状はモジュール定数の固定寸法 + `isPlayer` の青/赤2択。
8. HUD に小さく現在のペルソナ名を表示(任意)。
9. ドキュメント更新: 確定後 `GAME_DESIGN.md` に「§12 プレイヤータイプ」を新設。

> 選択画面のプレビューは、レーダーチャートに加えて**実際の3Dモデル(またはその
> シルエット)**を小さく表示すると、身長・体格・髪型などの身体的個性がひと目で
> 伝わる(§3.6/§3.7)。回転表示やチームカラー適用後の姿を見せられるとなお良い。

### 3.6 身体プロファイル(身長・体格・利き手)

能力(§3.2)に加え、各ペルソナに**身体的特徴**を持たせる。これは見た目の識別子で
あると同時に、ごく一部は軽いゲーム挙動にも繋がる(過剰にしない。バランスの主軸は
あくまで §3.2 の能力値)。

| ペルソナ | 身長 | 体格 | 利き手 | ゲームへの軽い影響(任意) |
|---|---|---|---|---|
| ① サンブラント | 1.88m | athletic | 右 | 高打点でサーブ/スマッシュがわずかに有利 |
| ② アガチ | 1.80m | athletic | 右 | 標準 |
| ③ ジョコヴィン | 1.88m | slim | 右 | リーチがわずかに広い(拾い) |
| ④ ニシゴオリ | 1.78m | slim | 右 | 低重心で初動がわずかに速い演出 |
| ⑤ ナダウ | 1.85m | stocky | **左** | **左利き**:フォア/バックが左右反転(クロスの軌道が変わる) |
| ⑥ フェデルン | 1.85m | athletic | 右 | 標準 |

**実装方針**:
- `heightM` は `CharacterEntity` の**全体縦スケール**(基準身長 ~1.83m に対する比)に変換。
  打点系の定数(`SERVE_HIT_HEIGHT`、スマッシュ成立高 `SMASH_MIN_HEIGHT` の判定基準)に
  身長比を小さく反映してもよい(任意・控えめに)。
- `build` は手足・胴の半径系定数(`TORSO_R`/`ARM_R`/`LEG_R`/`HIP_R`)のスケールに変換
  (slim=細い、stocky=太い)。**横太りでシルエットが変わる**のが識別に効く。
- `handedness === 'left'` は `player.ts` のスイング側(fore/back)判定を左右反転。
  描画は利き腕を反対側にするか、モデル全体を y 軸まわりに鏡像化する簡便策でもよい。
  → ナダウの「左利きの順クロス」という現実の個性が出る。

### 3.7 3Dモデルの外見とチームカラー(1P/2P)

**ねらい**: ①どのペルソナかが見た目で分かる、かつ ②自分と相手が**色で**即座に区別できる
(いわゆる 1Pカラー/2Pカラー)。この2つを両立するため、**色とシルエットの役割を分ける**:

- **チームカラー(1P/2P)= 主色**: ユニフォーム(胴・腰・脚)の基調色。
  自分=クール系(青)、相手=ウォーム系(赤)。これが最優先の識別。
  現状 `entities.ts` の `isPlayer ? 0x2255cc : 0xcc2222` を**チームパレット**として一般化:
  ```ts
  // constants.ts
  export const TEAM_PALETTE = {
    player:   { body: 0x2255cc, limb: 0x1a3f99, trim: 0x6699ff }, // 1P 青系
    opponent: { body: 0xcc2222, limb: 0x991a1a, trim: 0xff7766 }, // 2P 赤系
  }
  ```
- **シルエット = ペルソナの主識別子(色に依存しない)**: 身長・体格(§3.6)・髪型
  (`hair`)・袖の有無(`sleeves`)。色を変えても形で分かるのが強い。
  例:ナダウ=がっしり+ノースリーブ+長髪、アガチ=スキンヘッド(bald)、
  ジョコヴィン/フェデルン=ヘッドバンド、ニシゴオリ=細身+ショート。
- **アクセント色 = ペルソナ固有(チーム色と独立)**: ヘッドバンド/リストバンド/
  シューズ/ガット等の小物に乗せる。**青・赤に紛れない色**(白・黄・緑・橙・黒)を選び、
  チーム主色の上でも視認できるようにする。

| ペルソナ | hair | sleeves | アクセント色 | 見た目の要点 |
|---|---|---|---|---|
| ① サンブラント | short | sleeved | 白 (0xffffff) | 長身・正統派・白基調の小物 |
| ② アガチ | bald | sleeved | 橙 (0xff8a3d) | スキンヘッド+派手なアクセント |
| ③ ジョコヴィン | headband | sleeved | 黄緑 (0xc6ff3d) | 細身長身+ヘッドバンド |
| ④ ニシゴオリ | short | sleeved | 黄 (0xffd23d) | 小柄・細身・俊敏な印象 |
| ⑤ ナダウ | long | sleeveless | 緑 (0x33cc66) | がっしり+ノースリーブ+長髪(左利き) |
| ⑥ フェデルン | headband | sleeved | 黒 (0x222222) | 端正・ヘッドバンド・落ち着いた小物 |

> アクセントは「ペルソナの色」。同じナダウでも 1P なら青ユニ+緑小物、
> 2P なら赤ユニ+緑小物。**ユニ色=陣営、小物色+シルエット=ペルソナ**で二重に識別できる。

**`entities.ts` のパラメータ化(実装方針)**:
1. `CharacterEntity` のコンストラクタを
   `constructor(scene, opts: { team: Side; physique: PersonaPhysique; appearance: PersonaAppearance })`
   に変更(現状の `isPlayer: boolean` を置換)。`team` から `TEAM_PALETTE` を引く。
2. 寸法のモジュール定数(`HIP_Y`/`TORSO_R`/`ARM_R`/`LEG_R`/`THIGH_LEN`…)を
   **基準値**として残しつつ、`heightM` 比で縦、`build` で太さをスケールした
   ローカル値を組み立て時に算出する(マジックナンバーの基準は `constants.ts` に集約)。
3. 髪型・袖・小物をオプションのメッシュとして追加:
   - `headband`:頭部にトーラス(アクセント色)。`bald`:髪メッシュを省き頭頂を肌色のまま。
     `long`:後頭部に小さな箱/カプセル。`cap`:半球。
   - `sleeveless`:上腕メッシュを肌色に(袖なし表現)、`sleeved`:上腕をユニ色(`limb`)に。
   - リストバンド/シューズのライン等にアクセント色を点で効かせる。
4. 既存のスイング/走りアニメ(ピボット階層)を壊さないため、スケールは原則
   **ジオメトリ寸法**側に適用する(`group.scale` 一括だとラケット長や見えと物理 `REACH` が
   ずれる)。**見た目スケールと物理は分離**し、物理の `REACH` 等はペルソナの
   `speed`/身長補正(§3.4/§3.6)で扱う。

## 4. その他の改善アイディア(優先度つき)

ペルソナの戦略的価値を高める順に。`★優先度`(高=先にやる価値)。

### ★★★ 高(ペルソナと相乗効果が高い・比較的低コスト)
- **モメンタム/プレッシャー**(§1.4): ブレークポイント・ゲームポイントで品質に微小な
  変動(勝負強さ=隠しメンタル値)。攻撃型は普段強いがプレッシャーでブレ、グラインダーは
  安定…等、ペルソナに `mental` 隠し軸を持たせると相性ドラマが増す。
- **ラリーカウンター連動の優位**: 相手を左右に振った"走らせ距離"を蓄積し、オープンコートが
  実際に空く(=AI の戻り遅れ)ことを可視化。配球(§1.1)が報われる手応えを強化。
- **マッチ後スタッツ拡充**: ウィナー/エラーに加え、平均ラリー長・ネットポイント勝率・
  サーブ確率を表示。ペルソナごとの戦い方の違いが数字で見える。

### ★★ 中(深みは増すがコスト中〜大)
- **サーフェス選択**(§1.3): クレー(遅い/高バウンド=グラインダー有利)、
  グラス(速い/低い=サーバー有利)、ハード(中間)。`REST`/`BOUNCE_FRICTION`/球速減衰を
  サーフェス係数でスケール。ペルソナ相性に環境変数が加わり戦略が一段深くなる。
- **ネットプレー/ボレーの明確化**: 前進中の短いテイクバックのボレー、パッシング/ロブの
  読み合いを独立メカ化。サンブラント/フェデルンの個性が立つ。
- **AI のペルソナ別 AI ロジック**: 例 サンブラント AI はサーブ後に前へ詰める、
  ニシゴオリ AI はドロップ多用、ナダウ AI は深いトップスピンで粘る等、`ai.ts` の
  ショット選択重みをペルソナで変える(難易度=上手さ、ペルソナ=好み)。

### ★ 低(将来・任意)
- **ダブルス / 2人プレイ**(構成変更が大きい)。
- **トーナメント/キャリアモード**: ペルソナを選んで複数 AI を勝ち抜く。相性ゲームが活きる。
- **リプレイ/ハイライト**、**観客・音響の盛り上げ**演出。
- **モバイル/タッチ操作**対応。

     まず色とシルエット(身長・体格・髪型・袖)だけでも識別効果は十分大きい。
