## 4. その他の改善アイディア(優先度つき)

ペルソナの戦略的価値を高める順に。`★優先度`(高=先にやる価値)。

### ★★★ 高(ペルソナと相乗効果が高い・比較的低コスト)
- **モメンタム/プレッシャー**(§1.4): ブレークポイント・ゲームポイントで品質に微小な
  変動(勝負強さ=隠しメンタル値)。攻撃型は普段強いがプレッシャーでブレ、グラインダーは
  安定…等、ペルソナに `mental` 隠し軸を持たせると相性ドラマが増す。
  → **§5 でスタミナと絡めて具体化**(プレッシャー時の消耗・クラッチ回復)。
- **ラリーカウンター連動の優位**: 相手を左右に振った"走らせ距離"を蓄積し、オープンコートが
  実際に空く(=AI の戻り遅れ)ことを可視化。配球(§1.1)が報われる手応えを強化。
- **マッチ後スタッツ拡充**: ウィナー/エラーに加え、平均ラリー長・ネットポイント勝率・
  サーブ確率を表示。ペルソナごとの戦い方の違いが数字で見える。

### ★★ 中(深みは増すがコスト中〜大)
- **サーフェス選択**(§1.3): クレー(遅い/高バウンド=グラインダー有利)、
  グラス(速い/低い=サーバー有利)、ハード(中間)。`REST`/`BOUNCE_FRICTION`/球速減衰を
  サーフェス係数でスケール。ペルソナ相性に環境変数が加わり戦略が一段深くなる。
- **ネットプレー/ボレーの明確化**: 前進中の短いテイクバックのボレー、パッシング/ロブの
  読み合いを独立メカ化。サンブラント/フェデルンの個性が立つ。
- **AI のペルソナ別 AI ロジック**: 例 サンブラント AI はサーブ後に前へ詰める、
  ニシゴオリ AI はドロップ多用、ナダウ AI は深いトップスピンで粘る等、`ai.ts` の
  ショット選択重みをペルソナで変える(難易度=上手さ、ペルソナ=好み)。

### ★ 低(将来・任意)
- **ダブルス / 2人プレイ**(構成変更が大きい)。
- **トーナメント/キャリアモード**: ペルソナを選んで複数 AI を勝ち抜く。相性ゲームが活きる。
- **リプレイ/ハイライト**、**観客・音響の盛り上げ**演出。
- **モバイル/タッチ操作**対応。

## 5. スタミナシステムの戦略性強化(緩急を活かす消費モデル + ペルソナ精神力)

`★★★ 高`。配球(§1.1)・リスク管理と並ぶ「体力管理」軸(GAME_DESIGN §6/§11)を、
**「むやみに動く/強打し続ける」ことに代償**を与え、**緩急(休む・緩い球で繋ぐ)に報酬**を
与える形に拡張する。同時に**通常移動は軽く**、最低限のラリーは"微減"程度に抑えて
ストレスにしない。さらにペルソナの**隠しパラメータ(精神力)**で終盤の失速・勝負強さの
ドラマを出す。

### 5.1 ねらいと現状

- **現状**(`player.ts` `updateStamina(dt, sprinting)` / `constants.ts`):
  スタミナが減るのは**スプリント中だけ**(`STAMINA_SPRINT_DRAIN` 18/s)。
  非スプリント時は一律 `STAMINA_REGEN` 6/s で回復。→ 通常移動も強打も無コストで、
  「走り回る・全部フルチャージ」に体力的なデメリットがない。
- **したいこと**:
  1. 非スプリントでも**移動量(速さ)に比例**して少しずつ消費。
  2. **強いショット(チャージ/フラット/スマッシュ)ほど打球時に消費**。緩い球(スライス/
     ドロップ/ノーチャージ)はほぼ無消費。
  3. ただし**通常移動の消費は小さく**、「通常移動+適度な静止+ノーチャージで最低限の
     ラリー維持」なら**徐々に微減する程度**に収める(減りすぎてストレスにしない)。
  4. ペルソナの**精神力(隠し)**で、プレッシャー時の消耗・回復量に個性を出す。

### 5.2 提案する消費モデル(常時微回復 − 移動量比例消費 − 強打コスト)

現状の「スプリント時だけ減る」を、**常時の基礎回復から移動・強打のコストを差し引く
加算モデル**に置き換える(全て `constants.ts` に集約。マジックナンバーを散らさない規約に従う)。

```
毎フレーム(連続消費・回復):
  dStamina/dt = +STAMINA_REGEN_IDLE                         // 常時の基礎回復(速度0で全量)
              − STAMINA_MOVE_DRAIN_K · speed                // 移動量(現在の速さ m/s)に比例
              − STAMINA_SPRINT_EXTRA · [sprinting]          // スプリントの追加消費(瞬発の代償)
  ※ 移動・スプリント消費には既存のペルソナ倍率 mods.staminaDrainMul を乗算、
     基礎回復には mods.staminaRegenMul(× クラッチ回復 §5.5)を乗算する。

打球の瞬間(インパクト時に1回だけ差し引く、§5.3):
  stamina −= shotStaminaCost(shot) · mods.staminaDrainMul
```

**新定数(初期案・要チューニング)**:

```ts
// constants.ts(STAMINA_* の並びに追加)
export const STAMINA_REGEN_IDLE = 7    // /s 常時の基礎回復(速度0で全量効く)
export const STAMINA_MOVE_DRAIN_K = 1.4 // /s あたり(m/s)。drain = K · speed
export const STAMINA_SPRINT_EXTRA = 12  // /s スプリント時の追加消費
// 既存 STAMINA_REGEN(6) と STAMINA_SPRINT_DRAIN(18) は本モデルで置き換え(削除 or 移行)。
```

**バランス目標**(中立ペルソナ・倍率1.0 のときの net /s):

| 状態 | speed | net /s | 体感 |
|---|---|---|---|
| 静止して待つ | 0 | **+7.0** | しっかり回復(良い読み・ポジショニングの報酬) |
| 通常移動(歩行) | 5.5 | 7 − 1.4·5.5 = **−0.7** | ほぼ均衡=**微減**(ストレスなし) |
| やや速い詰め | 6.5 | **−2.1** | じわっと減る |
| スプリント | 8.0 | 7 − 1.4·8 − 12 = **−16.2** | 明確に消耗(現状 −18 と同程度) |

→ 「通常移動+適度な静止」を繰り返す限りスタミナはほぼ保たれ(微減)、
**走らされる・自分から走り回る・強打を連発する**ほど減る。緩急が効く。

### 5.3 強いショットの打球時消費

打球の瞬間に、ショットの"強さ"に応じてスタミナを1回差し引く。緩い球は実質無コスト。

```ts
// constants.ts
// ショット種ごとの基礎コスト(打球1回あたり)
export const SHOT_STAMINA_BASE: Record<ShotType, number> = {
  flat: 4, topspin: 3, slice: 1.5, lob: 1, drop: 1,
}
export const SHOT_STAMINA_CHARGE = 8       // · min(charge, 1) を加算(チャージ強打の代償)
export const SHOT_STAMINA_OVERCHARGE = 4   // · max(0, charge−1)/(CHARGE_MAX−1) を加算
export const SMASH_STAMINA_EXTRA = 4       // スマッシュ成立時の追加

// shotStaminaCost(shot) = SHOT_STAMINA_BASE[type]
//   + SHOT_STAMINA_CHARGE·min(charge,1) + SHOT_STAMINA_OVERCHARGE·overcharge率
//   + (スマッシュなら SMASH_STAMINA_EXTRA)
```

**コスト例**(中立):

| ショット | 例コスト | 戦術的意味 |
|---|---|---|
| ノーチャージのスライス/ドロップ | 1〜1.5 | 繋ぎ・緩急。ほぼ無消費でラリー維持できる |
| ノーチャージのトップスピン | 3 | 安定ラリーの主軸。低コスト |
| フルチャージのフラット | 4 + 8 = **12** | 決め球。連発は体力的に不可能 |
| オーバーチャージのフラット | 4 + 8 + 4 = **16** | ハイリスク・高コスト |
| フルチャージのスマッシュ | 4 + 8 + 4 = **16** | 決定機にだけ使う一撃 |

→ **「全球フルチャージ」は体力的に破綻**(10球で 120 消費 > タンク100)。
スライス/トップスピンで繋ぎ、チャンスだけ強打する**緩急の配分**が正解になる。
品質システム(§4.2/4.5 の「体勢が悪いと強打は自滅」)と二重に強打の乱用を抑える。

### 5.4 ストレスを生まないための調整(微減バランスの担保)

- **ポイント間回復**(`STAMINA_POINT_RECOVERY` 40)・**ゲーム間全回復**は維持。
  1ポイントの微減はポイント間に取り戻せるので、普通に打ち合う分には枯渇しない。
- **低スタミナの品質低下**(`calcStaminaFactor` / `STAMINA_LOW_THRESHOLD` 30%)は据え置き。
  消費モデルだけ変え、効き方(品質カーブ)は現状のままにして体感の地続き性を保つ。
- 数値は `scripts/measure.mjs`(§5.7)で**「ソフトに繋ぐだけのラリーは枯渇しない/
  走らされ+強打連発は明確に枯れる」**ことを確認しながら詰める。

### 5.5 ペルソナ隠しパラメータ:精神力(mental)とスタミナ特性

**方針**: 「スタミナの大きさ・効率」は**可視の `stamina` 軸**(レーダー §3.2、既存
`staminaMaxMul/DrainMul/RegenMul`)で表す。その上に、レーダーに出さない**隠しパラメータ
`mental`(精神力 1..5)** を1本足し、**プレッシャー耐性とクラッチ回復**を担わせる。
「スタミナの切れやすさ」は基本この2層(可視タンク × 隠し精神力)で表現する。

**`mental` の効果**:
- **クラッチ回復(時間回復ボーナス)**: 勝負強いタイプは基礎回復・ポイント間回復が増える。
  → 「勝負強いタイプはスタミナの時間回復量が増加」というご要望そのもの。
- **プレッシャー時の消耗**: 大事な場面(ブレーク/ゲーム/マッチポイント等)で、
  精神力が低いと**消耗が増え**(+狙い誤差も微増)、高いと**むしろ平常〜微強化**。

**導出(`constants.ts`、§3.4 の `personaModifiers` 隣に追加)**:
```ts
// mental ∈ {1..5} → 倍率(レーダー外の隠しパラメータ)
clutchRecoveryMul = 0.85 + 0.07·mental   // m5→1.20(回復+20%)  m1→0.92
pressureDrainMul  = 1.30 − 0.10·mental   // m5→0.80(プレッシャーで消耗↓)  m1→1.20(消耗↑)
```
適用:
```
基礎回復・ポイント間回復 ×= clutchRecoveryMul
プレッシャー p∈[0,1] のとき、移動・強打の消費 ×= 1 + (pressureDrainMul − 1)·p
   (高 mental は p が高いほど消費が減り、低 mental は増える)
   ※ 余力があれば品質の狙い誤差にも (pressureDrainMul−1)·p を小さく反映(メンタルでブレる)
```

**プレッシャー値 `p` の算出**(`core/scoring.ts` の `ScoreView` から `main.ts` が計算し、
`ControlContext` に `pressure: number` を足して各コントローラへ注入):
- マッチポイント / セット相当の決め所 → p ≈ 1.0
- ブレークポイント(レシーバーがゲーム奪取の1点前)/ ゲームポイント → p ≈ 0.7
- アドバンテージ・デュース → p ≈ 0.4
- 30-30 等の競り → p ≈ 0.25 / それ以外 → p ≈ 0
- サーバー/レシーバーで非対称にしてもよい(BP はレシーバー有利・サーバーに重圧)。

**各ペルソナの値(可視 stamina は §3.2 から再掲、mental は新規・隠し)**:

| ペルソナ | 可視 stamina | 隠し mental | キャラクター付け |
|---|---|---|---|
| ① サンブラント | 3 | 4 | ビッグマッチに強い。サーブで重圧をしのぐ |
| ② アガチ | 4 | 3 | ムラがあり感情的。乗ると強いが競り合いで揺れる |
| ③ ジョコヴィン | 5 | **5** | 鉄のメンタル。プレッシャーで逆に強く、回復も早い |
| ④ ニシゴオリ | 2 | **2** | 終盤に失速しやすく、競り合いで消耗(速さ・技巧で先行逃げ切り型) |
| ⑤ ナダウ | 5 | **5** | 不屈の闘志。長期戦・重圧に滅法強く回復力も高い |
| ⑥ フェデルン | 3 | 4 | 沈着。重要局面でも崩れにくい |

→ **ドラマの例**: ニシゴオリは速さ・技巧で序盤リードを奪うが、長いラリーと
ブレークポイントの連続で**失速して逆転される**。ジョコヴィン/ナダウは
**競れば競るほど強い**。プレイヤーは「自分の型なら短期で決めるか/粘って削るか」を選ぶ。

### 5.8 スタミナの可視化(プレイヤー近傍の円形ゲージ + 発汗エフェクト)

**現状の課題**: スタミナは画面**下部左右**のバー(`ui.ts` `hudStaminaPlayerBar`/
`hudStaminaOpponentBar`、`styles.css .hud-stamina`)で表示。ボールやキャラから視線が
離れた隅にあり、ラリー中は**見に来ないと分からない**。スタミナは新モデル(§5.2)で
常時動く重要情報になるため、**視線を外さず把握できる**見せ方に変える。

**方針 — 2層で表現**(数値的 × 直感的):

- **(A) 数値層 = キャラ近傍の円形ゲージ**: 残量を正確に・色で段階的に示す。
- **(B) エフェクト層 = 発汗など**: 数字を読まなくても「そろそろヤバい」が体感で分かる。

両者を**同じ閾値**(§5.2 の品質低下点 `STAMINA_LOW_THRESHOLD`=30%)に揃え、
「黄色く色づく → 汗をかき始める → 赤+大量の汗=品質が落ちる領域」と段階を一致させる。

#### (A) 円形ゲージ(キャラの頭上/足元に追従)

- **配置**: 既存の `renderer.worldToScreen()`(サーブラベルで使用中)でキャラの頭上
  (または足元)のスクリーン座標を求め、その位置に**DOM の円形ゲージ**を毎フレーム追従
  させる。`main.ts` がプレイヤー/AI の頭上ワールド座標を `worldToScreen` で変換し、
  スクリーン座標と残量を `HudView` 経由で UI に渡す。
- **形**: リング状アーク(ドーナツ)。残量に比例して**円弧が満ちる/欠ける**
  (満タン=全周、枯渇=ほぼ消える)。SVG の `stroke-dasharray`/`stroke-dashoffset`、
  または CSS `conic-gradient` で実装(外部アセット・ライブラリなしの規約に従う)。
  リング内に小さく**数値(%)**を出す(常時 or 低下時のみ)。
- **色(緑→黄→赤)**: 残量割合 `pct = stamina / effStaminaMax`(ペルソナで上限が変わるため
  **割合**で判定)で段階変化。境界は品質システムと整合させる:
  ```ts
  // constants.ts(可視化パラメータ)
  export const STAMINA_GAUGE_GREEN  = 0.60 // これ以上は緑(余裕)
  export const STAMINA_GAUGE_YELLOW = 0.30 // これ以上は黄(注意) = STAMINA_LOW_THRESHOLD/STAMINA_MAX
  //            これ未満は赤(品質低下が始まる危険域)
  ```
  緑→黄→赤は**境界で切り替え**でも、`pct` で hue を連続補間(120°緑→60°黄→0°赤)でも可。
  急減時(強打/スプリント)に一瞬**太く光らせる/振動**させると代償が伝わる(§5.6-6 と統合)。
- **自分 / 相手の差**: 自分のゲージは大きく/明るく、相手は小さめ/控えめ。
  リング枠を**チームカラー**(1P青/2P赤、§3.7 `TEAM_PALETTE`)で縁取ると、
  残量色(緑/黄/赤)と陣営色が両立する(枠=陣営、塗り=残量)。
- **視認性の配慮**: キャラに完全に重ねると邪魔なので**頭上に少し浮かせる**。
  打球・スイング中はゲージを少し淡くしてプレイの邪魔をしない。
- **既存の下部バーの扱い**: プレイ中の主表示はこの近傍ゲージへ移す。下部バーは
  **撤去**するか、ごく小さな補助(数値のみ)に縮小して画面をすっきりさせる。

#### (B) 発汗・疲労エフェクト(ざっくり伝える層)

`PlayerView.stamina` は描画に渡っており(`CharacterEntity.update(dt, view)` で参照可能)、
**追加配線なしで** 3D 側の疲労表現を駆動できる。

- **発汗パーティクル**: `pct` が `STAMINA_SWEAT_START`(例 0.45、黄色域の少し手前)を
  下回ると、頭・肩から**汗滴のスプライト**を散らし始める。`pct` が下がるほど
  **放出レートと量を増やす**(赤域で最大)。小さな滴メッシュ/ビルボードをプールで
  使い回す(`effects.ts` の既存プール方式に倣う。`entities.ts` の `CharacterEntity` に
  発汗エミッタを持たせるのが配線的に最短)。
  ```ts
  // constants.ts
  export const STAMINA_SWEAT_START = 0.45 // この割合未満で発汗開始
  export const STAMINA_SWEAT_MAX_RATE = 14 // /s(pct→0 で最大放出レート)
  ```
- **補助的な疲労サイン**(任意・軽く):
  - **荒い息**: 低スタミナ時、胴体をゆっくり上下に脈動(呼吸)させる。
  - **重い構え**: 待機姿勢をやや前傾・低くする(`relaxArms`/`updateLegs` に微調整)。
  - **ポイント間の仕草**: 非常に低いときはポイント決着後に「膝に手」のポーズ。
  - **自分が危険域のときだけ**画面隅に**薄いビネット/わずかな彩度低下**を出し、
    プレイヤー本人に体感的な警告を与える(相手のときは出さない)。
- これらは**閾値を (A) と共有**するので、「黄色く色づく=汗ばむ」「赤=滝の汗+息切れ」と
  ゲージとエフェクトが一致し、数字を読まなくても状況が分かる。

#### 実装タッチポイント(§5.8)

1. `types.ts`: `PlayerView` に正規化残量 `staminaPct`(0..1)を追加(色・発汗の判定用)。
   `HudView` に各キャラの**スクリーン座標**と `staminaPct`(+チーム)を追加。
2. `main.ts`: 毎フレーム、両キャラ頭上の `worldToScreen` を計算して `HudView` に詰める。
3. `ui/ui.ts` + `ui/styles.css`: 円形ゲージ要素(SVG/conic-gradient)を追加し、
   渡された座標へ追従・残量で色とアークを更新。下部バーは撤去 or 縮小。
4. `render/entities.ts`: `CharacterEntity` に発汗エミッタ+疲労ポーズを追加
   (`view.staminaPct` で駆動)。滴はプールで使い回す。
5. `render/effects.ts`(任意): 画面ビネットなどポストな演出を足す場合の受け皿。
6. `constants.ts`: `STAMINA_GAUGE_*`/`STAMINA_SWEAT_*` を集約。
7. `scripts/visualcheck.mjs`: 高/中/低スタミナのスクリーンショットを撮り、
   色段階・発汗・視認性(ボールと被らないか)を確認。
