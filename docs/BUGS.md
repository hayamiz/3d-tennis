## BUG-001 — サーブをノーバウンドで打ち返すと次の着地でフォルト判定が走る

- **ステータス**: 🟢 修正済み(commit: 後述)
- **発見日**: 2026-06-13
- **影響範囲**: `src/core/rally.ts`(`RallyJudge`)/ サーブレシーブ全般

### 修正内容
`RallyJudge.onEvent` の `hit` 処理で、**サーブフェーズ中にサーバー以外(レシーバー)が
打った時点で `phase` を `'rally'` に遷移**させるようにした(サーバーを `reset` で
`this.server` に保持)。これにより、ノーバウンド返球後の最初の着地は通常ラリーの
イン/アウト判定で処理され、サービスボックス判定(フォルト)が誤って走らなくなった。
回帰テスト 2 件を `tests/rally.test.ts` に追加(ノーバウンド返球の着地が in なら決着なし/
out なら out)。全 80 テスト・型チェック・ビルド green。

### 再現手順
1. サーブを打つ。
2. レシーバーがサーブをノーバウンド(ボレー)で打ち返す。
3. 打ち返した球が次に着地する。

### 期待
- サーブが正規(サービスボックス内に着地)であれば、ノーバウンドで返球された
  時点で通常ラリーに移行し、以降は通常ラリーの規則で判定されるべき。

### 実際
- 返球後の最初の着地で、サービスボックス判定(フォルト判定)が走ってしまう。
  本来通常ラリーのイン/アウト判定であるべき着地が、サーブのフォルトとして
  処理されてポイントが誤って決着する。

### 原因(推定)
`RallyJudge` のフェーズ遷移 `'serve' → 'rally'` は、サーブ球の**バウンス**を
契機に `_handleServeBounce` の中でしか行われない(`rally.ts` の
`_handleServeBounce` 内の `this.phase = 'rally'`)。

ノーバウンドで返球された場合、サーブ球は一度も地面でバウンドしないため、
`phase` は `'serve'` のまま残る。一方で `onEvent` の `'hit'` イベントで
`lastHitter` はレシーバーに更新される。結果、返球が次に着地したときの
`bounce` イベントで `_handleBounce` が `phase === 'serve'` を見て
`_handleServeBounce` を(レシーバーを打者として)呼び出し、サービスボックス
判定=フォルト判定を誤って適用してしまう。

### 修正の方向性(メモ)
- サーブ後にレシーバーが `hit` した時点(= サーブフェーズ中に server 以外の
  `hit` イベントを受けた時点)で `phase` を `'rally'` に遷移させる。
  → ノーバウンド返球を正しく「サーブ成立 + ラリー開始」として扱える。
- 修正後は `scripts/diag.mjs`(`?debug`)でサーブ→ノーバウンド返球→着地の
  判定ログを確認し、フォルトが出ないこと・通常ラリー判定になることを検証する。

## BUG-002 — ドロップショットを AI が見送り(下がって構えて届かず)2バウンドで失点する

- **ステータス**: 🔴 未着手
- **発見日**: 2026-06-13
- **影響範囲**: `src/gameplay/ai.ts`(戦術スタンス `stanceGoalZ` の baseline 構え位置)/
  短い球・ドロップショットの処理。**戦術スタンス機能(GAME_DESIGN §7.1)導入による回帰**。

### 再現手順
1. AI(奥コート)とラリーする。
2. AI を一度ベースライン深くに追い込む(深い球を打つ)。
3. 続けて**ドロップショット**(ネット際に短く落ちる球)を AI 側へ打つ。
4. AI がドロップに前へ詰めず、ボールがバウンドしたのを実質見送って 2 バウンドし、
   こちら(プレイヤー)の `doubleBounce` ポイントになる。

### 期待
- 短いドロップに対しては AI が**前へ詰めて**バウンド直後(または前)に拾いに行くべき。
  少なくとも着地点まで出て、届く限り返球を試みる。

### 実際
- AI が着地点より**手前(自陣ベースライン側=深い位置)に下がって構え**、
  ほとんど跳ねずに失速するドロップに届かず、2 バウンドして失点する。
  ログ(hard / 相手=ジョコヴィン)で確認:
  - `t=8.59` `stance baseline (score=0.39 tend=0.1 short=0.58 landingDepth=3.12)` —
    着地はネットから 3.12m の短い球。スコア 0.39 < 閾値 0.62 で baseline スタンス。
  - 直前の `t=7.53` の自打球は `hitZ=-10.76`(ベースライン深く)。AI はそこから前進が必要。
  - 以降 **shot イベントが出ないまま** `t=10.89` `verdict doubleBounce → player`。
    = AI は一度も打てていない。

### 原因(分析)
`stanceGoalZ`(`src/gameplay/ai.ts`)が baseline スタンスで、着地点より常に
`AI_BASELINE_DROPBACK`(1.6m)**深い(ネットから遠い)位置**を構え目標にしている:

```
return sign * (depth + AI_BASELINE_DROPBACK)  // depth = |着地点 z|
```

この「着地点より後ろに下がって、バウンド後の上がり際で打つ」設計は、
**通常の打球がバウンド後さらに自陣奥へ進む**ことを前提にしている(BUG なし)。
しかし**ドロップショットはバウンド後ほとんど前進せずネット際で失速して死ぬ**ため、
前提が成立しない:
- ドロップは着地点(depth≈3.12)付近で 2 バウンド目を迎える。
- AI の構え目標は `depth+1.6 ≈ 4.72`、すなわち**死ぬ球より 1.6m 奥**。
  ボールはそこまで来ないので、待っていても永久にリーチに入らない。
- さらに今回は AI が深く(`z≈-10.76`)にいたため、前進で約 6m 走る必要があり、
  反応遅延 + 移動では、下がり込む 1.6m の余分も相まって 2 バウンド目までに
  リーチへ届かない。

要するに、**baseline の「着地点より後退」オフセットを短い球・低バウンド球にも
無条件で適用している**のが原因。スタンス機能導入前は着地点そのものを目標にして
いたため(より前で)拾えていた挙動が、後退オフセットで悪化した回帰。
スコア上もベースライナー(`netRushTendency=0.1`)は短い球(short=0.58)でも
閾値 0.62 に届かず baseline のままで、ドロップに対して下がってしまう。

### 修正の方向性(メモ)
- baseline の後退オフセットを**短い球・低バウンド球には適用しない**(または着地点を
  超えて後退しないようクランプ)。例: 着地が一定の浅さ(例 `SERVICE_LINE_Z` 以内)、
  あるいはドロップ相当(着地が浅く `incomingSpeed` が小さい / 予測のバウンド後到達が
  浅い)のときは、構え目標を**着地点〜やや前**にして前進で拾わせる。
- より原理的には、`stanceGoalZ` の後退は「バウンド後に自陣奥へ伸びる球」に限定し、
  伸びない球(ドロップ)では前へ。`predictLanding` をバウンド後まで延長して
  ボールが実際に到達する深さ・好打点高さを見て構え位置を決めると恒久的に解決する
  (IMPROVEMENTS の旧§4★★★(a) 高精度版に相当)。
- 検証は `scripts/diag.mjs`(`?debug`)で「深い球→ドロップ」の配球を再現し、
  ドロップに対し AI が前進して shot イベントを出す(=拾う)ことを確認する。
